# /write-query - 编写优化的 Hive SQL

> 如果你看到不熟悉的占位符或需要查看已连接的工具，请参阅 [CONNECTORS.md](../../CONNECTORS.md)。

根据自然语言描述编写 Hive SQL 查询，遵循最佳实践。

## 使用方法

```
/write-query <你需要的数据描述>
```

## 工作流程

### 1. 理解需求

解析用户的描述，识别以下要素：

- **输出列**：结果应包含哪些字段？
- **筛选条件**：哪些条件限制了数据（时间范围、用户群体、状态）？
- **聚合操作**：是否有 GROUP BY 操作、计数、求和、平均值？
- **连接**：是否需要合并多个表？
- **排序**：结果如何排序？
- **限制**：是否有 top-N 或采样要求？

### 2. 发现 Schema（已连接数据仓库时）

如果数据仓库 MCP 服务器已连接：

1. 根据用户描述搜索相关表
2. 检查列名、类型和关系
3. 检查影响性能的分区键或聚簇键
4. 查找可能简化查询的预建视图或物化视图

### 3. 编写查询

遵循以下最佳实践：

**结构：**
- 当查询有多个逻辑步骤时，使用 CTE（WITH 子句）提高可读性
- 每个逻辑转换或数据源对应一个 CTE
- 为 CTE 取描述性名称（如 `daily_signups`、`active_users`、`revenue_by_product`）

**性能：**
- 生产环境查询绝不使用 `SELECT *`——只指定需要的列
- 尽早筛选（将 WHERE 子句尽可能靠近基表）
- 善用分区过滤器（尤其是日期分区）
- 大结果集子查询优先使用 `EXISTS` 而不是 `IN`
- 使用适当的 JOIN 类型（内联接正确时不要用左联接）
- 可以用 JOIN 或窗口函数时避免使用相关子查询
- 注意防止爆炸式连接（多对多）
- Hive 的 `SORT BY`（区内排序）、`ORDER BY`（全局排序）、`DISTRIBUTE BY`、`CLUSTER BY` 按需使用
- 对于大表聚合，考虑使用 `GROUP BY` 的优化写法避免数据倾斜

**可读性：**
- 为非显而易见的逻辑添加注释解释"为什么"
- 使用一致的缩进和格式
- 为表取有意义的首字母别名（不要只用 `a`、`b`、`c`）
- 每个主要子句单独占一行

**Hive 特定优化：**
- 使用 `EXPLAIN` 分析查询执行计划
- 合理设置 `mapjoin` 优化小表 join
- 考虑 `bucket` 表优化大规模 join
- 使用 `INSERT OVERWRITE` 而非多次 INSERT

### 4. 展示查询

提供：

1. **完整查询**——带语法高亮的 SQL 代码块
2. **简要说明**——每个 CTE 或部分的作用
3. **性能说明**（如有）——预期成本、分区使用情况、潜在瓶颈
4. **修改建议**——如何调整以适应常见变化（不同时间范围、不同粒度、额外筛选条件）

### 5. 提供执行选项

如果已连接数据仓库，主动提出运行查询并分析结果。如果用户想自己运行，查询已准备好可复制粘贴。

## 示例

**简单聚合：**
```
/write-query 统计最近 30 天各状态的订单数量
```

**复杂分析：**
```
/write-query 留存率分析——按注册月份对用户分组，然后展示在注册后 1、3、6、12 个月仍活跃（至少有一次事件）的用户比例
```

**性能关键：**
```
/write-query 我们有一张大表按日期分区。找出最近 7 天事件数量排名前 100 的用户及其最近一次事件类型。
```

## 技巧

- Hive 没有 `DISTINCT ON`（PostgreSQL 特有），需要用窗口函数实现类似功能
- 日期函数常用 `DATE_ADD`、`DATEDIFF`、`YEAR`、`MONTH`、`DAY` 等
- 字符串函数常用 `CONCAT`、`SUBSTR`、`LENGTH`、`TRIM` 等
- 聚合函数注意 `COUNT(DISTINCT col)` 在大数据量时的性能问题，可考虑用 `GROUP BY` 代替
- 如果知道表名就包含进去——否则 Claude 会帮你查找
- 说明查询是否需要幂等性（可安全重复运行）还是一次性查询
- 对于 recurring 查询，说明是否需要参数化日期范围
