# 2026 第一周 SQL 案例逐案沉淀矩阵

生成日期：2026-06-06

## 使用方式

本矩阵把 26 个 SQL 案例逐条压缩成 `write-query` 技能可吸收的知识单元。核心不是保存工单 SQL，而是回答：

- 用户这样问时，是什么业务场景？
- 主业务事实在哪张表？
- 主表不够时补哪张表？
- 关键过滤、JOIN、审计点是什么？
- 哪些硬编码、临时表、一次性枚举不能进入技能？

后续真正改技能时，优先把“主表路由规则”写入 `ROUTING.md` / `TABLE_INDEX.md`，把“补表规则”写入 `FIELD_BACKFILL.md`，把“审计规则”写入 `RULES.md`，只有复杂且高复用的流程才进入 `verified-cases`。

## 逐案沉淀

### 1. 销售品到期与赠金/折扣参数匹配

- 源文件：`第1题_吉敏_20260605_XQGZ2025111001789脚本_1.sql`
- 用户可能会问：根据一批号码查指定销售品是否在档、销售品到期时间、赠金/折扣参数。
- 场景标签：`附件号码清单`、`销售品在档`、`销售品参数`、`移动号码`
- 主表路由规则：这类需求的事实不是订单动作，而是“账期在档销售品”，主表应选 `014 优惠资料表 / ads_yz_rpt_comm_cm_msdisc_final`；若用户只给号码，先用 `069 全业务资料表 / dwm_yz_tb_comm_cm_all_final` 按 `acc_nbr + par_month_id` 补 `serv_id`。
- 补表规则：销售品参数不在 014 时，候选补 `summary_ods_day_city.rpt_comm_cm_msparam`，按 `serv_id + prod_offer_id + param_code` 取参数值；该表需先进入待核对。
- 关键字段/过滤：`prod_type=30`、`limit_date >= 账期月底`、`prod_offer_code in (...)`、`row_number() over(partition by serv_id order by limit_date desc)` 取最晚到期销售品。
- 应沉淀到：`ROUTING.md` 销售品在档场景；`FIELD_BACKFILL.md` 销售品参数补字段；`RULES.md` 附件号码核数和一号多销售品去重审计。
- 不应沉淀：具体销售品编码 `YD0203-*`、具体参数编码、附件表名、工单临时表名。

### 2. 骑士卡项目账目项收入明细

- 源文件：`第1题_吉敏_20260605_XQGZ2025111201472脚本_2.sql`
- 用户可能会问：筛一批特定客户/产品的号码，输出账目项、科目路径、税后收入明细。
- 场景标签：`移动号码圈定`、`科目级收入明细`、`项目收入`
- 主表路由规则：先用 `069 全业务资料表` 按客户名、产品分类、产品类型圈定 `serv_id`；收入明细事实应路由到 `048 全量科目级收入 / dwm_srhx_src_income_list_mon`，不要用 069 的 `fee` 替代科目级收入。
- 补表规则：无固定补表；主表输出已包含科目、账目项、收入来源。
- 关键字段/过滤：`yd_prod_type2`、`cust_name like`、`prod_type=30`、`serv_id in (...)`、收入阈值可由用户指定。
- 应沉淀到：`ROUTING.md` 项目号码 -> 科目级收入；`TABLE_INDEX.md` 校准 048 生产表名漂移。
- 不应沉淀：客户名“达能”、`subst_id=4050`、收入阈值 10/30 作为通用口径。

### 3. 产权客户编码匹配直销客户与机构

- 场景标签：`客户补字段`、`产权客户`、`直销客户`、`机构补字段`

## 按技能文件归并后的实施视图

### ROUTING.md 优先补充

- 销售品在档与参数：用户说“销售品到期、是否办理、折扣/赠金参数”时，先 069 取 `serv_id`，主查 014，在必要时补参数表。
- 科目级收入明细：用户说“账目项、科目路径、收入来源、税后收入明细”时，主查 048，不用 069 的 `fee` 替代。
- 直销客户/产权客户：用户说“产权客户编码、直销客户编码、普通商客”时进入客户映射场景，不要直接猜 069 字段。
- 商机转化：用户说“电信到家/场景化/政企OAO 商机转化、入湖时间、sa_at”时，先识别智能平台外部线索表，再号码关联 069。
- 揽装积分/电信到家积分：积分事实优先查 `ads_yz_score_all_list`，不是收入表或佣金表。
- 主副卡：副卡查主卡直接从 069 的 `zk_acc_nbr`。
- IMSI/ACMB 折扣率：属于号码属性/特性补字段场景，不是 069 直接字段。
- 城域网普通商客、客户经理 CRM、固话使用时长、实收金额、移机订单、临街商铺地址，均需新增或待核对路由入口。

### FIELD_BACKFILL.md 优先补充

- 地址层级：`serv_addr_id -> dwd_yz_addr_final.id -> addr_id_7/addr_id_6/parentid -> addr`；地址 ID 过长时优先统一转字符，避免 decimal 强转漏数。
- 销售品名称：`prod_offer_code -> dws_crm_cfguse.dws_offer.offer_name`，限制 `city_id=200`。
- 销售品参数：`serv_id + prod_offer_id + param_code -> param_value`，参数表待核对。
- CRM 工号：`sales_code -> dws_crm_cfguse.dws_staff.staff_code -> staff_account`，历史多版本取最新。
- IMSI：`serv_id -> dws_crm_cust.dws_prod_inst_attr.prod_inst_id`，`attr_id=200000103`。
- ACMB 折扣率：`serv_id -> tb_pre_cm_attr_all_mon`，`attr_id=500049004`。
- 主副卡：`acc_nbr -> zk_acc_nbr`。
- 直销客户/普通商客：`cust_code/cust_nbr -> 直销客户表`，`vip_flag` 口径待核对。

### RULES.md 优先补充

- 附件清单必须保留序号，输出后核对输入行数/输出行数；如果业务允许一对多，要说明膨胀原因。
- 多账期打横必须明确起止账期，避免 `>=` 无结束条件。
- 历史多版本维表/员工表必须有 `row_number` 去重，并明确倒序取最新。
- 时间窗匹配必须说明是否取全部命中、最近一条、最早一条。
- 地址 ID 类型转换要审计，禁止默认把所有 `serv_addr_id` 强转 decimal。
- 号码、身份证等敏感字段输出要脱敏。
- CTAS 中全局 `order by` 不应作为默认排序手段，除非结果规模和需求确认。

- 智能平台商机累计积分/出账/佣金：外部线索表无 `serv_id`、号码去重、累计区间。
