# SC-010 揽装人认领责任田调整

## 适用

用户要按“揽装人认领规则”找出当前责任田应调整的号码清单，例如：

- 单移 / 3C / 指定号码范围内，按揽装人应落入的责任田认领服务。
- 对比号码当前划小责任田与揽装人认领规则责任田，输出不一致的号码。
- 需要输出目标责任田、当前责任田、双方包区 / 营服 / 分局，以及当前划分规则、认领规则。

## 不适用

- 只查普通揽装人姓名、工号、网点或有效性：看 111/112/113。
- 只按责任田统计收入或客户：优先看收入/客户主表，再按责任田字段汇总。
- 只查号码当前划小归属：069 当前字段已够用时不必补认领规则表。

## 主路径

```text
069 全业务资料表
  -> 124 揽装人认领规则表（按 sales_code 找目标责任田）
  -> 125 服务当前划小规则表（按 serv_id 补当前划分/认领规则）
  -> 按目标责任田与当前责任田差异筛选
```

## 步骤

### 1. 圈定号码底座

主表用 069 `dwm_yz_tb_comm_cm_all_final`；历史账期按 069 日/月表规则选择。

常用字段：

- 服务与号码：`serv_id`、`acc_nbr`
- 入网时间：`open_date`
- 揽装人：`sales_id`、`sales_code`
- 当前责任田：`grid_id`、`grid_code`、`grid_name`
- 当前包区 / 营服 / 分局：`area_id`、`branch_id`、`subst_id`
- 副卡：`is_vice_card`

常见筛选：

- 账期分区用 `par_month_id`，入网业务月用 `date_format(open_date,'yyyyMM')`。
- 产品范围（如移动、3C、单移）需来自用户、稳定产品维表或已确认口径；不要把单个案例的产品 ID 直接固化。
- 在网 / 未拆机类需求通常加 `is_cancel_user=0`，但仍以用户需求为准。

### 2. 补揽装人认领规则

补表 124：

```sql
left join ads_yz_sales_grid_rule_list r
  on base.sales_code = r.sales_code
```

常用输出：

- `r.grid_code` 目标责任田编码
- `r.grid_id` 目标责任田 ID
- `r.grid_name` 目标责任田名称
- `r.grid_type_name` 目标责任田类型
- `r.area_id` 目标包区 ID
- `r.branch_id` 目标营服 ID
- `r.subst_id` 目标分局 ID

如果生产使用清洗表或别名（如 `ads_yz_sales_grid_rule_list_cl`），落 SQL 前按现网口径确认表名。

### 3. 补当前划分规则

补表 125：

```sql
left join dwd_yz_jyfx_serv_grid_final g
  on base.serv_id = g.serv_id
```

常用输出：

- `g.subst_rule` 当前划小局向 / 机构划分规则
- `g.grid_rule` 当前责任田认领 / 划分规则
- `g.subst_date`、`g.grid_date` 规则日期

### 4. 筛出需要调整的号码

通用判断：

- 必须命中揽装人认领规则：`r.grid_id is not null`
- 当前责任田与目标责任田不一致：`coalesce(base.grid_id,0) <> coalesce(r.grid_id,0)`
- 包区不一致，或包区都为空 / 无效但营服不一致时，列入待处理。

示例条件：

```sql
where r.grid_id is not null
  and coalesce(base.grid_id,0) <> coalesce(r.grid_id,0)
  and (
    ((base.area_id not in (0,-1) or r.area_id not in (0,-1))
      and coalesce(base.area_id,0) <> coalesce(r.area_id,0))
    or (base.area_id in (0,-1) and r.area_id in (0,-1)
      and coalesce(base.branch_id,0) <> coalesce(r.branch_id,0))
  )
```

业务排除条件（例如剔除套餐规则、省政企、副卡、需求单号码）必须来自本次需求或已确认口径，不作为本场景无条件硬规则。

## 输出字段

建议输出：

| 业务字段 | 来源 |
|---|---|
| 服务标识 | 069 `serv_id` |
| 接入号 | 069 `acc_nbr` |
| 揽装人编码 | 069 `sales_code` |
| 目标责任田编码 / ID / 类型 | 124 `grid_code/grid_id/grid_type_name` |
| 当前责任田 ID / 名称 | 069 `grid_id/grid_name` |
| 目标包区 / 营服 / 分局 | 124 `area_id/branch_id/subst_id` |
| 当前包区 / 营服 / 分局 | 069 `area_id/branch_id/subst_id` |
| 入网时间 | 069 `open_date` |
| 当前划分规则 / 认领规则 | 125 `subst_rule/grid_rule` |
| 是否副卡 | 069 `is_vice_card` |

## 风险审计

- `sales_code` 可能不唯一，124 同一工号可能多条规则；必须核对 `sales_code` 去重后粒度。
- 124 表名可能存在正式表与清洗表别名；落 SQL 前确认现网表名。
- 125 只补规则来源，不替代 069 当前责任田字段。
- 包区、营服、分局字段含义要区分当前归属和目标认领归属，输出时建议加 `cur_` / `target_` 前缀。
- 剔除“套餐规则”、省政企、副卡、需求单号码等属于业务口径参数；不要从一个案例扩大为全局硬规则。

having count(*) > 1
limit 20;
